Production Systems & Applied AI

Semantic Search for Domain Terminology

Problem: Executive search platform needed semantic search across profiles, job descriptions, and company data. Standard full-text search (Elasticsearch BM25) and fuzzy matching couldn't handle domain-specific terminology—"CEO" vs "Chief Executive" vs "C-suite" vs "top executive" needed to match intelligently.

Tried: Elasticsearch with synonym dictionaries (became unmaintainable), fuzzy matching (too many false positives), BM25 tuning (couldn't capture semantic similarity).

Solution: Integrated sentence-transformers (all-MiniLM-L6-v2) with vector similarity search. Embedded documents offline, stored vectors in Postgres with pgvector extension. Query time: embed search term, cosine similarity, rerank with BM25 for hybrid results.

Trade-offs:

  • Added 40ms avg latency (acceptable for our use case)
  • Cold start penalty on model loading (~2s, solved with Lambda provisioned concurrency)
  • Storage cost: ~120MB vectors for 500k documents
  • Model drift risk: retraining pipeline runs monthly on user feedback

Architecture: Python API (FastAPI) → Inference on AWS Lambda (1GB memory) → Postgres + pgvector → .NET API layer

Outcome: Search relevance improved from 62% to 89% (measured via user clickthrough). Cost: $120/month inference + $40/month storage. Shipped in 6 weeks with team of 3.

Anomaly Detection for Data Pipeline Quality

Problem: Multi-tenant data ingestion pipeline processed ~2M records/day. Manual data quality checks (SQL queries checking nulls, outliers, schema violations) became unmaintainable. Percentile-based thresholds broke when client data distributions changed.

Tried: Rule-based validation (too rigid), statistical thresholds (false positives during seasonal changes), manual review (couldn't scale past 50 clients).

Solution: Built anomaly detection using isolation forests on feature distributions. Tracked 40+ metrics per client (record counts, null rates, value distributions, schema changes). Trained per-tenant models on 90 days historical data, flagged records with isolation forest anomaly scores above the 95th percentile of baseline scores.

Why ML here: Data patterns varied wildly across clients—healthcare vs finance vs manufacturing. Deterministic rules couldn't capture legitimate variation vs actual problems.

Trade-offs:

  • False positive rate: 3–5% (acceptable with human review queue)
  • Training time: 20min nightly per tenant (batched, not blocking)
  • Required 90-day warm-up period for new clients
  • Model explainability limited (isolation forest internals aren't obvious)

Architecture: Postgres → Python batch job (scikit-learn) → Feature extraction → Isolation forest training → Redis cache (model predictions) → Alert pipeline

Outcome: Reduced data quality incidents from ~12/week to ~2/week. Caught 3 critical schema breaks before they hit production pipelines. Cost: $200/month compute + $80/month Redis.

Multi-Tenant API Platform (No AI)

Problem: Built API platform serving 200+ clients with isolated data, varied SLAs (some require <100ms p99, others tolerate 2s), and different rate limits per tenant. Needed horizontal scaling, fault isolation, cost control.

Architecture:

  • .NET 6 API layer (ASP.NET Core) behind AWS ALB
  • PostgreSQL (RDS Multi-AZ) with read replicas
  • Redis (ElastiCache) for session state, rate limiting, cache
  • Kubernetes (EKS) with horizontal pod autoscaling
  • Tenant isolation via database schemas + connection pooling
  • Distributed tracing (Jaeger) + metrics (Prometheus/Grafana)

Scaling challenges solved:

  • Connection pool exhaustion: moved to pgbouncer, reduced connections from 2000+ to 200
  • Cache stampede on popular queries: added probabilistic early expiration (randomly expire entries slightly before TTL to distribute regeneration load)
  • Hot tenant problem: implemented per-tenant circuit breakers
  • Cost control: spot instances for non-critical workloads (saved ~40% compute cost)

Why no AI: Deterministic caching, load balancing, and rate limiting solved the problems. Adding ML would've introduced latency and complexity without measurable benefit.

Outcome: Served 50M requests/day at p99 latency of 145ms. Uptime: 99.95% (outages mostly AWS-related). Cost: ~$8k/month infrastructure (optimized from initial $14k).
